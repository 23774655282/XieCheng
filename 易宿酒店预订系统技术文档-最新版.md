# 易宿酒店预订系统 — 技术文档

本文档对易宿酒店预订系统（Hotel-Booking）进行完整技术说明，涵盖架构、技术栈、数据模型、接口、前端实现、部署与运维等细节，供开发与维护参考。

---

## 1. 项目概述

### 1.1 产品定位

易宿酒店预订系统是一套面向真实业务场景设计的**全栈 B2C 酒店预订平台**，面向三类角色：

- **C 端用户**：搜索酒店、查看房型与评价、在线预订、支付、退款、个人中心与收藏。
- **商户（B 端）**：酒店与房型管理、订单与入住管理、退款审核、商户入驻申请。
- **平台管理员**：酒店/商户/房型审核、上下线、退款纠纷介入。
- **架构描述**：项目采用**React + Vite + Tailwind CSS**构建前端单页应用，配合**React Router**做多页面路由与懒加载；后端基于**Node.js + Express**提供RESTful API，使用**MongoDB + Mongoose**管理数据模型，通过**JWT**实现无状态认证。系统集成了**高德地图JS API**实现酒店地图展示与点聚合，并结合**二维码支付、AI 智能搜索**，支持按预算、城市和偏好进行自然语言搜索，给出更智能的房型推荐。整体架构支持同源部署：生产环境下由**Express**托管前端构建产物及静态资源，并代理API与上传服务，简化部署与运维。

### 1.2 系统架构概览

```
                    ┌─────────────────────────────────────────┐
                    │            Browser (SPA)                  │
                    │  React 19 + Vite + React Router + Axios   │
                    └─────────────────┬───────────────────────┘
                                       │ HTTP/HTTPS
                                       ▼
                    ┌─────────────────────────────────────────┐
                    │     Node.js (Express 5)                  │
                    │  REST API / 静态资源 / 上传文件           │
                    └─────────────────┬───────────────────────┘
                                       │
              ┌────────────────────────┼────────────────────────┐
              ▼                        ▼                         ▼
    ┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
    │    MongoDB      │    │  高德 Web API    │    │     二维码	     │
    │  (Mongoose ODM) │    │ (逆地理/地理编码) │     │     (支付)      │
    └─────────────────┘    └─────────────────┘    └─────────────────┘
```

- **前后端分离**：前端 SPA 通过 `VITE_BACKEND_URL` 访问后端 REST API。
- **同源部署**：生产环境由 Express 托管 `client/dist`，并代理 `/api`、`/uploads`，避免跨域。
- **无状态后端**：认证依赖 JWT（Bearer Token），无服务端 Session 存储。

## 4. 数据模型（MongoDB / Mongoose）

### 4.1 User

- **_id**: String，主键（可与 Clerk 等外部 ID 对齐）。
- **username**, **avatar**, **birthday**: 展示与资料。
- **phone**: String，必填、唯一，用于登录与验证码。
- **passwordHash**: 必填，bcrypt 等哈希。
- **email**: 可选，用于找回密码等。
- **role**: `'user' | 'merchant' | 'admin'`，默认 `'user'`。
- **merchantApplicationStatus**: `'none' | 'pending' | 'approved' | 'rejected'`，商户申请状态。
- **favoriteHotels**: [ObjectId]，收藏酒店。
- **recentSerachCities**: [String]，最近搜索城市（服务端可写）。
- **resetCode**, **resetCodeExpiresAt**: 找回密码验证码。
- **loginCode**, **loginCodeExpiresAt**: 验证码登录。
- **timestamps**: createdAt, updatedAt。

### 4.2 Hotel

- **name**, **nameEn**, **address**, **contact**, **city**: 基础信息。
- **owner**: ObjectId → User，商户。
- **latitude**, **longitude**: 可选，地图展示。
- **starRating**: 1–5，**openTime**: 开业时间。
- **hotelIntro**: 酒店介绍文案。
- **status**: `pending_audit | pending_list | approved | rejected`。
- **rejectReason**: 驳回原因。
- **images**, **hotelExteriorImages**, **hotelInteriorImages**: 图片 URL 数组。
- **doorNumber**, **totalRoomCount**: 再审核补充。
- **supplementSubmitted**: 是否已提交再审核。
- **applicantName**, **applicantPhone**, **licenseUrl**, **starRatingCertificateUrl**: 预审/入驻材料。
- **nearbyAttractions**, **promotions**: 兼容旧字段。

### 4.3 Room

- **hotel**: ObjectId → Hotel。
- **roomType**: String（如 Single Bed, Double Bed）。
- **pricePerNight**: Number。
- **amenties**: [String]，设施列表。
- **images**: [String]，图片 URL。
- **isAvailable**: Boolean。
- **promoDiscount**: Number | null，0–100 优惠百分比。
- **roomCount**: Number，库存。
- **status**: `pending_audit | approved`，上架审核。

### 4.4 Booking

- **user**, **room**, **hotel**: ObjectId 引用。
- **roomQuantity**: Number，预订间数。
- **checkInDate**, **checkOutDate**: Date。
- **totalPrice**, **guests**: Number。
- **guestName**, **guestEmail**, **guestPhone**, **guestRemark**: 联系人信息。
- **status**: `pending | confirmed | cancelled`；**cancelledBy**, **cancelReason**。
- **paymentMethod**, **isPaid**, **isCompleted**, **hasReview**。
- **paymentConfirmToken**, **paymentConfirmTokenExpiresAt**: 扫码支付确认。
- **refundRequested**, **refundReason**, **refundStatus** (`pending | approved | rejected`)，**refundReviewedAt**, **refundReviewedBy**。
- **refundPlatformReviewRequested**, **refundPlatformReviewReason**：用户申请平台介入。
- **merchantRequestedPlatformIntervention**：商家请求平台介入。
- **stayStatus**: `pending_checkin | checked_in | checked_out`。

### 4.5 Review

- **user**, **hotel**, **booking**: ObjectId；**rating**: Number；**comment**: String；**images**: [String]。

### 4.6 其他集合

- **MerchantApplication**：商户入驻申请（用户申请成为商户）。
- **HotelPreReviewApplication**：商户新增酒店预审单（执照、内外景图等），与 MerchantApplication 一起在管理员「预审核（执照审核）」中展示。
- **HotelSupplementEdit**：酒店补充信息再审核。
- **RoomEditApplication**：房型**修改**审核（改价、改数量、上架/下架等）；已有房间的修改在管理员「再审核（酒店信息）」中随酒店一并审核。
- **Room.status = pending_audit**：**新增房型**待审核，在管理员「房间上架申请」中单独审核通过后变为 approved。
- **RegisterVerification**：注册/登录验证码等（若使用）。

## 5. 认证与鉴权

### 5.1 JWT 流程

- 登录/注册成功后，后端返回 JWT 与 user 信息，前端通过 `setAuthFromLogin({ token, user })` 直接更新 Context 并写入 `localStorage`（key 如 `token`），再延迟一帧 `navigate`，避免状态未提交时跳转导致新页仍显示未登录。
- 请求需鉴权接口时，在 Header 中携带：`Authorization: Bearer <token>`。
- 后端 `authMiddleware`：解析 Bearer、`jwt.verify(token, JWT_SECRET)`、查 User、挂载 `req.user`。

### 5.2 角色与路由保护

- **user**：可访问 C 端所有页面及个人中心、我的预订、收藏等。
- **merchant**：在 **user** 基础上可访问 `/owner/*`（酒店与房型管理、订单、退款审核）。
- **admin**：可访问 `/admin/*`（酒店/商户/房型审核、退款纠纷介入）。

前端通过 `AppContext` 中的 `role`、`isOwner`、`isPlatformAdmin` 控制导航与入口展示；敏感操作依赖后端再次校验（如 `adminOnly` 中间件）。

### 5.3 管理员中间件

- `adminOnly`：在 `authMiddleware` 之后使用，校验 `req.user.role === 'admin'`，否则 403。

## 6. 核心 API 说明

### 6.1 认证 `/api/auth`

- `POST /register/send-code` — 发送注册验证码。
- `POST /register` — 注册。
- `POST /login` — 账号密码登录。
- `POST /login/send-code` — 发送登录验证码。
- `POST /login/by-code` — 验证码登录。
- `POST /forgot-password/request` — 请求重置验证码。
- `POST /forgot-password/verify` — 校验验证码。
- `POST /forgot-password/reset` — 重置密码。

### 6.2 商户与预审核 `/api/merchant`

- `GET /applications` — 预审核（执照审核）列表，支持 `status`（pending/approved/rejected）、`page`、`limit`（10 或 20），返回 `applications`、`totalCount`、`page`、`limit`。
- `POST /applications/:id/approve` — 通过商户申请。
- `POST /applications/:id/reject` — 驳回商户申请（body 含 rejectReason）。
- `POST /applications/hotel-add/:id/approve` — 通过商户新增酒店预审单。
- `POST /applications/hotel-add/:id/reject` — 驳回商户新增酒店预审单。

### 6.3 用户 `/api/users`

- `GET /` — 当前用户信息（需鉴权）。
- `PATCH /profile` — 更新资料（含头像上传）。
- `GET /favorites` — 收藏列表。
- `POST /favorites/:hotelId` — 添加收藏。
- `DELETE /favorites/:hotelId` — 取消收藏。
- `POST /store-recent-search` — 保存最近搜索城市（需鉴权）。

### 6.4 酒店 `/api/hotels`

- `GET /public` — 公开酒店列表（分页等）。
- `GET /public/cities` — 城市列表。
- `GET /public/map` — 地图视野内酒店（minLat/maxLat/minLng/maxLng，可选 city、starRating）。
- `GET /public/search` — 公开搜索（如 q=关键词）。
- `GET /public/:id` — 单酒店详情（含房型列表）。
- `POST /` — 商户注册酒店（鉴权）。
- `GET /owner/list`, `GET /owner/:id` — 商户自己的酒店；支持 `page`、`limit`（10/20），返回 `totalCount`、`page`、`limit`。
- `PUT /owner/:id/reapply`, `PUT /owner/:id/supplement` — 再审核、补充材料。
- `GET /my`, `PUT /my` — 当前商户酒店信息与更新。
- `GET /audit`, `GET /audit/:id` — 管理员再审核（酒店信息）列表与详情；支持 `status`、`page`、`limit`（10/20），返回 `totalCount`、`page`、`limit`。
- `POST /audit/approve`, `POST /audit/reject` — 审核通过/驳回；**offline**、**restore**、**supplement-edits** 等。

### 6.5 房型 `/api/rooms`

- `GET /` — 分页列表。查询参数：`page`、`limit`、`destination`（城市/名称）、`promo`（1–100 整数，表示展示**优惠力度 ≥ promo%** 的房间）、`lat`、`lng`（附近筛选）。返回 `rooms`、`total`、`hasMore`。
- `POST /smart-search` — 智能搜索（body 含 query、预算等，内部可调 AI 与聚合）。
- `GET /:id` — 单房型详情。
- `POST /` — 商户创建房型（鉴权 + 上传 images）。
- `GET /owner`, `GET /owner/:id` — 商户房型列表与详情；支持 `hotelId`、`page`、`limit`（10/20），返回 `totalCount`、`page`、`limit`。
- `PUT /:id`, `DELETE /:id` — 更新、删除。
- `PATCH /:id/room-count`, `PATCH /:id/promo-discount` — 库存与优惠。
- `GET /admin/pending-adds` — 房间上架申请列表（**仅新增房型** Room.status=pending_audit/approved）；支持 `status`、`page`、`limit`（10/20），返回 `totalCount`、`page`、`limit`。已有房间的上架/修改在「再审核（酒店信息）」中处理。
- `POST /admin/pending-adds/:id/approve`, `POST /admin/pending-adds/:id/reject` — 通过/驳回新增房型上架。

### 6.6 预订 `/api/bookings`

- `POST /check-availability` — 检查房型在日期段内是否可订（房间数、库存）。
- `POST /book` — 创建订单（鉴权）。
- `GET /user` — 当前用户订单列表。
- `GET /hotel` — 当前商户酒店订单列表。
- `POST /stripe-payment` — 创建 Stripe 支付会话。
- `POST /confirm-payment` — 扫码/回调确认支付（含 paymentConfirmToken）。
- `GET /:id/pay-qr` — 获取支付二维码等。
- `PATCH /:id/cancel` — 取消订单。
- `PATCH /:id/complete` — 用户标记完成（驱动评价入口）。
- `POST /:id/request-refund` — 用户申请退款。
- `PATCH /:id/supplement-refund-reason` — 补充退款原因。
- `PATCH /:id/review-refund` — 商户审核退款（同意/拒绝）。
- `POST /:id/request-platform-refund-review` — 用户申请平台介入。
- `POST /:id/merchant-request-platform-intervention` — 商户请求平台介入。
- `PATCH /:id/stay-status` — 商户更新入住状态（待入住/已入住/已退房）。

### 6.7 评价 `/api/reviews`

- `POST /` — 创建评价（鉴权，可上传最多 3 张图）。
- `GET /hotel/:hotelId/stats` — 酒店评分与条数统计。
- `GET /hotel/:hotelId` — 酒店评价列表。
- `GET /me` — 当前用户历史评价。
- `DELETE /:reviewId` — 删除自己的评价。

### 6.8 高德 `/api/amap`

- `GET /regeo?lng=&lat=` — 逆地理（GCJ-02 坐标 → 省市区、详细地址）。服务端用 `AMAP_KEY` 请求高德，避免浏览器 CORS 与 Key 暴露。

## 7. 前端技术细节

### 7.1 构建与分包（Vite）

- **入口**：`index.html` → `src/main.jsx`，挂载 `#root`。
- **manualChunks**：将 `react`、`react-dom`、`react-router-dom` 打为 `vendor-react`；`react-hot-toast`、`react-datepicker` 打为 `vendor-ui`，以利缓存与并行加载。
- **chunkSizeWarningLimit**: 600（KB），避免大 chunk 警告。
- **自定义插件**：构建结束后将各 chunk 大小写入 `dist/bundle-size.json`，便于性能分析（如 ?perf=1）。

### 7.2 全局状态（AppContext）

- **userInfo**, **role**, **merchantApplicationStatus**, **isAuthenticated**, **isOwner**, **isPlatformAdmin**, **authChecked**：用户与权限。
- **setAuthFromLogin**：登录接口返回的 `{ token, user }` 直接写入 Context 与 localStorage，用于登录成功后立即展示已登录状态。
- **recentSearchRecords**, **searchCity**, **addRecentSearch**, **clearRecentSearch**：最近搜索（含 localStorage 持久化）。
- **rooms**, **fetchRooms**：首页等房间列表与分页。
- **fetchUser**：登录后或刷新时拉取用户信息并计算角色。
- **navigate**, **getToken**：路由与取 Token；Axios 的 baseURL 为 `import.meta.env.VITE_BACKEND_URL`。

### 7.3 样式与响应式

- **Tailwind 4**：通过 `@import "tailwindcss"` 与 `@theme` 使用；自定义 `--color-primary`、`--color-secondary`。
- **字体**：Google Fonts 引入 Outfit、Playfair，正文 Outfit，标题可选用 Playfair。
- **全局**：`html` 平滑滚动、取消 tap 高亮；`body` 最小高度、防止横向滚动；`#root` 弹性列布局；部分组件使用 `min-w-0`、`transition` 保证缩放与滚动时布局稳定。

### 7.4 性能优化

- **路由级懒加载**：除 Home 外均 `lazy()` + `Suspense`，统一 Loading 占位。
- **触底加载**：AllRooms、HotelDetail 房型列表使用 `IntersectionObserver` 监听底部哨兵元素，触底请求下一页或展开更多房型。
- **图片懒加载**：列表与卡片中的房间/酒店缩略图使用 `loading="lazy"`、`decoding="async"`。
- **虚拟列表**：酒店详情页房型列表、AllRooms 等使用 `@tanstack/react-virtual` 仅渲染视口内项，降低 DOM 数量。
- **轮播**：首页 Hero 轮播支持多图与无缝循环，可配合性能监控（PerfContext、carouselPerf）做首屏分析。

### 7.5 地图与地理

- **高德 JS API**：前端通过 `utils/loadAmap.js` 动态加载高德地图 JS API（Key 来自 VITE_AMAP_KEY）；旅行地图页（TravelMap）与首页小地图（HomeMap）使用 AMap，支持点聚合、点击聚类放大、点击酒店定位；聚合逻辑在 `utils/mapCluster.js`（仅当标记在当前缩放下会重叠时合并为一簇）。
- **地理编码与逆地理**：`utils/amap.js` 提供地理编码、WGS84→GCJ02；逆地理优先走服务端 `/api/amap/regeo`，避免 CORS 与 Key 暴露。

### 7.6 AI 智能搜索

- 前端 `/ai-hotel` 与智能搜索流程：用户输入自然语言（如预算、城市、偏好），请求后端 `POST /api/rooms/smart-search`；后端使用 OpenAI 兼容接口（如 DeepSeek）解析意图并组装查询条件，结合 MongoDB 聚合（如按折扣后价格筛选）返回推荐房型，前端展示在 SmartSearchResults 或对话流中。

## 8. 后端技术细节

### 8.1 启动与数据库

- **顺序**：先 `await connectDB()`，再 `app.listen(PORT)`；若 MongoDB 连接失败则 `process.exit(1)`，避免无库可用。
- **MONGO_URI**：在 `.env` 中配置，可含认证（username/password）、authSource、副本集等；生产建议使用独立数据库用户与强密码。

### 8.2 静态与缓存头

- **/uploads**：`express.static(..., { maxAge: "30d" })`，浏览器缓存 30 天，减轻重复访问图片流量。
- **client/dist**：带 hash 的 JS/CSS 等设为 `Cache-Control: public, max-age=31536000, immutable`；`index.html` 及 SPA 回退响应设为 `no-cache`，保证发版后能拿到新入口。

### 8.3 文件上传（Multer）

- 房间图：`uploads/rooms`，单次最多 4 张。
- 酒店图：`uploads/hotels`、商户申请/预审：`uploads/merchant-apply` 等；不同路由使用不同 multer 配置（fields/array）。
- 上传后 URL 由 `PUBLIC_URL` 或 `req.protocol + req.get("host")` 拼出，确保生产环境为公网域名而非 localhost。

### 8.4 定时任务

- **取消超时未支付订单**：启动时执行一次，之后每 60 秒执行一次；逻辑在 `cancelExpiredUnpaidBookings`（如 15 分钟内未支付则置为 cancelled）。

### 8.5 支付与 Webhook

- **Stripe**：创建支付会话、重定向或二维码；Webhook 使用 `express.raw({ type: 'application/json' })` 单独挂载，验证签名后更新订单 `isPaid` 等状态。
- **扫码确认**：`paymentConfirmToken` 一次性 token，前端支付成功页带 token 调用 `POST /confirm-payment` 完成确认。

### 8.6 价格与收入逻辑

- **预订总价**：按房型 `pricePerNight` 与 `promoDiscount` 计算折扣后单价，再乘以晚数与房间数；退款通过 `refundStatus === 'approved'` 等标识，商户收入统计会排除已同意退款的订单。

## 10. 部署与运维

### 10.1 生产构建

- 前端：`cd client && npm install && npm run build`，产物在 `client/dist`。
- 后端：`cd server && npm install`，无需单独构建。

### 10.2 进程管理（PM2）

- 推荐在服务器上使用 PM2 守护 Node 进程，例如：  
  `pm2 start server.js --name hotel-api` 或使用 `ecosystem.config.cjs`。
- 发版后：`git pull` → 前端 `npm run build` → 后端 `pm2 restart hotel-api`。

### 10.3 同机部署（Express 托管前端）

- 将 `client/dist` 放在 `server` 同级的 `client/dist`，`server.js` 中已配置：
  - 存在 `clientDist` 时托管静态资源并 SPA 回退；
  - 不存在时仅输出 "API is running..."，便于仅跑 API 的场景。

### 10.4 依赖说明

- 后端需安装 `compression` 才能启用 gzip（`npm install compression`）；若未安装，可注释掉 `app.use(compression())` 避免启动报错。
- 前端若使用虚拟列表、地图、AI 结果页，需保证 `@tanstack/react-virtual`、高德 JS API（通过 loadAmap 动态加载）等已配置，否则构建或运行会报错。

## 11. 安全与合规要点

- **密钥**：JWT_SECRET、高德 Key、数据库密码等仅放在服务端环境变量，不提交进仓库。
- **鉴权**：所有涉及用户/商户/订单/酒店的写操作与敏感读操作均经 `authMiddleware` 或 `adminOnly` 校验。
- **输入**：对目的地、搜索关键词等做 trim 与长度限制；文件上传限制类型与数量，避免恶意上传。
- **HTTPS**：生产环境建议全站 HTTPS，以使用浏览器定位、支付等能力。
